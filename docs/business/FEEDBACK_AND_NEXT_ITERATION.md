## 一、用户反馈收集渠道与周期

### 1.1 收集渠道

| 渠道 | 说明 | 覆盖角色 | 触发机制 |
|------|------|----------|----------|
| **产品内嵌反馈入口** | 投票详情页底部「反馈」小字链接，点击弹出迷你表单（分类单选 + 文字描述 ≤200 字） | 投票参与者 / 发起者 | 每次进入投票详情页自然暴露，无主动推送打扰 |
| **飞书团队群反馈贴** | 在团队群内固定一条「投票工具反馈收集」话题帖，置顶；成员可在帖子下回复使用问题 | 全体团队成员 | 上线后由云起手动发布，每月顶一次保持可见 |
| **定期回访问卷** | 飞书表单问卷， 5 道选择题 + 1 道开放题，2 分钟可完成 | 抽样 20 人（覆盖发起者/参与者/旁观者） | 上线后第 1 周 / 第 4 周各推送一次 |
| **数据埋点（静默收集）** | 前端关键行为打点上报（创建耗时、投票耗时、WS 断开频率、错误率），不使用第三方 SDK，后端着陆到 PG 日志表 | 全量用户（匿名聚合） | 自动，无需用户操作 |
| **栖梧/寻错巡检反馈** | 技术团队在运维中发现的性能瓶颈、错误日志异常、降级触发频次 | 技术视角 | 每周同步至云起 |

### 1.2 收集周期

| 阶段 | 时间窗口 | 活动 |
|------|----------|------|
| **上线首周** | Day 1–7 | 数据埋点首轮聚合分析 + 回访问卷（第 1 次）+ 团队群反馈帖开放 |
| **稳定运行期** | 第 2–4 周 | 每周末汇总反馈帖 + 埋点周报；第 4 周推送回访问卷（第 2 次） |
| **持续迭代期** | 第 5 周起 | 每月一次反馈汇总（来自内嵌入口 + 埋点 + 团队帖），输出月度《反馈简报》 |
| **按需** | 随时 | 栖梧/寻错提交严重 bug 或降级告警后 24h 内云起响应，判断是否需要紧急修复 |

### 1.3 反馈量预估与处理带宽

| 来源 | 预估月反馈量 | 处理方式 |
|------|------------|----------|
| 产品内嵌入口 | 5–15 条 | 云起每周筛选归类，P0 级立即响应 |

## 二、反馈分类框架

### 2.1 三级分类体系

```
反馈
├── 🐛 Bug
│   ├── B-P0 阻断：核心流程不可用（投票无法创建/提交/查看）
│   ├── B-P1 严重：功能可用但数据/状态异常（票数不准、状态不同步）
│   └── B-P2 一般：UI瑕疵、文案错误、非关键路径小缺陷
│
├── 💡 功能建议
│   ├── S-P0 必须：无此功能则核心场景无法覆盖（MVP 遗漏）
│   ├── S-P1 重要：显著提升效率或覆盖高频场景
│   └── S-P2 锦上添花：低频率但合理的长尾需求
│
└── ✨ 体验优化
    ├── U-P0 阻塞体验：用户明确表示反感或误操作频发
    ├── U-P1 明显摩擦：可用但使用过程有抱怨（多一步、慢一点）
    └── U-P2 精益求精：视觉/动画/微文案优化
```

### 2.2 判定矩阵

| 用户反馈 | 分类判断 | 优先级判断 |
|----------|----------|------------|
| "点提交没反应，一直转圈" | 🐛 Bug | B-P0 — 核心流程阻断 |
| "票数显示 3 票但实际只有 2 人投了" | 🐛 Bug | B-P1 — 数据不一致 |
| "能不能加个改票功能" | 💡 功能建议 | S-P2 — 范围外（本期明确不做） |
| "希望能指定谁能投票" | 💡 功能建议 | S-P1 — 定向投票人，高频需求 |
| "提交按钮太小了经常点错" | ✨ 体验优化 | U-P1 — 明显摩擦 |
| "柱状图能不能加个动画" | ✨ 体验优化 | U-P2 — 精益求精 |

### 2.3 反馈流转流程

```
用户提交反馈
    │
    ▼
[分类 + 优先级标注] ← 云起
    │
    ├── B-P0 / S-P0 ──→ 立即拉群（云起 + 栖梧 + 寻错）→ 24h 内排期修复
    │
    ├── B-P1 / S-P1 / U-P1 ──→ 纳入下个迭代 Planning
    │
    └── B-P2 / S-P2 / U-P2 ──→ 进入 Backlog 池，3 票以上升级
```

## 三、基于验收中 5 条存疑 AC 的潜在改进方向

> 以下 5 条均来自业务验收报告 Go/No-Go 判定的 ⚠️ 存疑项。因均在 P1/P2 级别，不构成本轮 Go 阻断，但在用户真实使用中最可能暴露，需优先关注。

### 存疑项 1：网络异常 UX 偏离 PRD（AC-001-10）

| 维度 | 描述 |
|------|------|
| **当前状态** | 创建投票网络异常时：按钮文案变为 `"发布中..."`（非 loading 旋转动画）；错误提示为页面内嵌红色区块（非 toast 浮层）；表单数据保留不清空（✅ 符合预期） |
| **用户影响** | ① 按钮文案变化容易被忽略，用户可能以为系统在正常处理；② 内嵌错误区块在移动端小屏可能被键盘遮挡；③ 无统一 toast 组件导致不同页面错误提示样式不一致 |
| **改进方向** | ① 引入统一 Toast 组件（成功/错误/警告），全局挂载到视口顶层，z-index 最高；② 创建投票按钮增加旋转加载动画（`<Spinner>` + 禁用态）；③ 网络异常 toast 自动 3s 消失 + 手动可关闭 |
| **建议迭代** | v1.2 — 作为全局 UI 组件基础设施，同步收敛全站错误提示样式 |
| **验收对照** | AC-001-10 → 达到 PRD 规定的 loading 旋转 + toast + 表单保留三个完整预期 |

---

### 存疑项 2：实名模式头像缺失 + 投票人明细展示方式（AC-003-2 / AC-006-2）

| 维度 | 描述 |
|------|------|
| **当前状态** | 实名模式下：① ECharts tooltip hover 仅显示投票人**文本姓名**，无头像；② 已结束投票的投票人明细仍为 hover 浮层方式，而非 PRD 要求的「不可折叠常驻展示」 |
| **用户影响** | ① 纯文本姓名在 10+ 人团队中辨识度差（同名、仅凭文字定位慢）；② 投票结束后用户期望**一眼看到谁投了哪个选项**，而非逐个 hover 翻阅——这是投票结果「可审计性」的核心体验 |
| **改进方向** | ① 接入飞书头像资源（飞书开放平台 `/image/v4/avatar/:user_id`），实名模式 tooltip 增加 24px 圆形头像 + 姓名；② 已结束投票每个选项下方展开**不可折叠**的投票人头像墙（横向排列，头像 32px，hover 显示姓名 Tooltip）；③ 匿名模式保持不变（后端字段级过滤 `getVotersMap → {}` 已正确） |
| **建议迭代** | v1.2 — 优先级 P1（头像为视觉增强，投票人常驻展示为核心体验偏差修复） |
| **验收对照** | AC-003-2 + AC-006-2 → 头像可见 + 已结束投票人明细不可折叠常驻 |

---

### 存疑项 3：WS 断开重连细节未验证（AC-003-7）

| 维度 | 描述 |
|------|------|
| **当前状态** | 前端 `useSocket` 配置了 `reconnection: true` + `reconnectionAttempts: 10` + 指数退避 `[1000ms, 30000ms]`；`NetworkBanner` 黄色横幅文案与 PRD 一致；重连后触发 `fetchDetail()` 全量拉取。但 useSocket/useNetworkStore 完整实现文件不在验收范围内，以下细节未确认：① 指数退避实际算法；② 断线期间 UI 是否禁用投票提交；③ 连续重连失败 10 次后的降级态 |
| **用户影响** | 弱网环境（地铁、电梯）下用户可能经历：① 连续提交失败但无明确提示；② 黄色横幅反复闪烁引起焦虑；③ 重连后图表抖动（乐观更新残留 + 全量覆盖冲突） |
| **改进方向** | ① 补充 useSocket/useNetworkStore 验收测试（栖梧 + 寻错）；② 断线期间禁用「提交投票」按钮并提示「网络不可用，请稍后重试」；③ 重连后先清空 `optimisticCounts` 再 `fetchDetail`，避免数据重叠；④ 连续重连失败 ≥5 次后降级为手动刷新模式（「点击刷新」按钮替代自动轮询） |
| **建议迭代** | v1.2 — 弱网体验增强，优先级 P1 |
| **验收对照** | AC-003-7 → 弱网全链路表现符合 PRD §5.3 WS 断线处理要求 |

### 存疑项 4：手动结束投票端到端流程未完整验证（AC-004-1）

| 维度 | 描述 |
|------|------|
| **当前状态** | 后端 `closeVote` 通道已验证（`FOR UPDATE` 锁行 + 权限校验 + WS 广播）；前端 `VoteDetail.tsx` 中「结束投票」按钮 + 确认弹窗 UI 结构已确认存在（通过 CSS Module `.dialogCancel/.dialogConfirm` 推断），但完整交互链未在验收范围内验证 |
| **用户影响** | 若前端权限判断逻辑有误，可能造成：① 发起者看不到结束按钮（无法结束投票）；② 非发起者误看到结束按钮（越权操作，虽后端兜底拒绝但体验怪异） |
| **改进方向** | ① 补充 VoteDetail.tsx 中 `isCreator` 判断逻辑验收（应与 `vote.creator_id === current_user_id` 一致）；② 手动结束后前端增加 2s 过渡动画（柱状图淡出「最终结果」标题淡入），降低页面突变感；③ 手动结束后广播消息中附带 `closed_by: 'manual'` 字段 + 发起者姓名，供其他在线用户展示「由 XXX 手动结束」 |
| **建议迭代** | v1.2 — 优先补充验收（P0），动画/通知增强为 P2 |
| **验收对照** | AC-004-1 → 发起者手动结束全流程（UI + 后端 + WS 推送）一致 |

### 存疑项 5：最终结果页 & 匿名过滤依赖未审查的 VoteDetail.tsx（AC-006 系列）

| 维度 | 描述 |
|------|------|
| **当前状态** | 后端数据聚合 + API 字段级过滤（`getVotersMap: anonymous → {}`) 已确认正确；前端 `VoteDetail.tsx` 中最终结果柱状图、投票人明细渲染、匿名模式隐私保护渲染均不在验收范围 |
| **用户影响** | 这是用户的核心终态体验——投票结束后他们只会看到最终结果页。若此处渲染有偏差，会严重削弱对工具的信任：① 匿名投票下前端意外展示投票人身份（虽后端数据已过滤，但前端缓存或错误处理可能泄露）；② 最终结果柱状图未标注「最终」字样，用户不确定结果是否已锁定 |
| **改进方向** | ① 补充 VoteDetail.tsx 最终结果页全量验收（需覆盖：柱状图数据源、匿名过滤、已结束标签、总票数统计）；② 最终结果页增加「数据锁定」视觉标识（🔒 图标 + 灰色背景 + 「最终结果」标题）；③ 匿名模式已结束页增加二次确认文案：「本次投票已结束，投票人身份信息已永久隐藏」 |
| **建议迭代** | v1.2 — 优先补充验收（P0），视觉标识为 P2 |
| **验收对照** | AC-006-1/AC-006-2/AC-006-3 → 最终结果页完整覆盖 |
