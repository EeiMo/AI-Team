# 项目交付物清单

> **起草人**：云起📋（项目经理）  
> **对齐版本**：8 人全栈团队架构  
> **适用范围**：从需求到运维的全生命周期交付物管理  
> **核心原则**：交付物分两层交付——下层「保命」（必须交付），上层「增值」（可选，但加分）。

## 阶段一：需求与立项

|------|------|

| 维度 | 说明 |

## 阶段五：安全

### 5.1 安全审计报告 —— ⭐ 必须

| 维度 | 说明 |
|------|------|
| **内容** | OWASP Top 10 检查（SQL 注入 / XSS / CSRF / 认证绕过 / 敏感信息泄露）、中间件安全版本扫描、依赖漏洞检查 |
| **受众** | 客户安全团队 + 凌霜/流光（修复）+ EeiMoo |
| **负责人** | **知微**（安全） |
| **AI 价值评估** | 💡 **安全审计是 AI 绝对不可替代的核心工作**。AI 可做 SAST/DAST 扫描输出格式报告，但**攻击面识别、风险定级、绕过测试思路**（如水平越权、JWT 签名校验缺失、SSRF 路径）需要渗透测试思维。AI 会漏报、会误判 |
| **价值** | 💡 **团队的终极价值护城河**。一次安全漏洞足以毁掉客户信任 |

### 5.2 渗透测试报告 —— ⭐ 必须

| 维度 | 说明 |
|------|------|
| **内容** | 手动渗透测试记录：鉴权绕过测试、水平越权尝试、会话劫持测试、敏感数据泄露检查 |
| **受众** | 客户安全团队 + 凌霜 |
| **负责人** | 知微 |
| **AI 价值评估** | **完全不可替代**。渗透测试需要「攻击者心态」，AI 不具备创造性攻击思路和上下文推理 |
| **价值** | 这是对安全审计的「实战检验」 |

### 5.3 数据观察计划 —— 可选增值

| 维度 | 说明 |
|------|------|
| **内容** | 上线后的安全监控指标（异常登录频率、异常 API 请求模式、敏感数据访问日志） |
| **受众** | 长夜（配置监控）+ 知微（分析）|
| **负责人** | 知微 |
| **AI 价值评估** | AI 可辅助异常检测，但**阈值定义、告警响应策略**需要安全判断 |
| **价值** | 安全不是一次性动作，持续观察才有效 |

## 高频问题 FAQ

**Q：这么多文档，项目的「三件套」是什么？**

A：优先级金字塔——
1. 🥇 **PRD + 架构文档 + 测试报告**（缺一个就敢上线？敢的团队大概率出问题）
2. 🥈 **安全审计 + 部署 Runbook**（上了生产后出事）
3. 🥉 其它一切（锦上添花）

**Q：AI 时代，我们的核心竞争力在哪？**

A：交付物中 **标 💡 的部分**——也就是 AI 会写但写不好的东西。AI 能写出 80 分的代码，但：
- 80 分的架构可能在一年后让团队重构
- 80 分的测试可能漏掉最关键的边界 Case
- 80 分的安全审计可能让客户数据裸奔

**我们的价值是「判断力」**：知道哪里该妥协、哪里不能妥协、出了问题怎么兜底。

**Q：所有交付物都必须有人手工校对吗？**

A：分三层——
- **AI 初稿 + 人审**：代码、测试用例、API 文档、设计初稿
- **纯手工完成**：安全渗透测试、架构设计选择、需求优先级判定、回滚方案
- **AI 主导 + 人验收**：文档格式整理、数据可视化、模板填充

**Q：哪些交付物最容易被 AI 替代？**

A：代码生成（CRUD 类型）、文档模板填充、测试用例骨架。这些用 AI 加速没问题，但**必须有人类 review**。

**Q：客户最看重哪些交付物？**

A：按客户类型分——
- **技术型客户**：安全审计报告 + 架构文档 + API 文档
- **业务型客户**：PRD（验收条件）+ 业务验收报告 + 数据观察报告
- **运维型客户**：部署 Runbook + 运维手册 + CI/CD 配置
