### 1.1 项目背景

团队在日常协作中频繁需要快速决策：技术方案二选一、团建地点投票、迭代优先级排序等。当前依赖聊天群内「接龙 + 手动数票」，存在以下痛点：

- **效率低下**：发起人逐条翻聊天记录手工统计，10 人以上团队统计耗时 >5 分钟
- **易出错**：手动统计遗漏、重复计数时有发生
- **缺乏隐私选项**：敏感议题（如团队满意度）无法匿名表达，导致沉默螺旋
- **无时间约束**：讨论无限拖延，缺乏「截止时间」的紧迫感

本工具提供一套**轻量、即时、团队内闭环**的投票能力，嵌入现有飞书工作流，零学习成本。

### 1.2 项目目标

| 指标 | 目标值 | 衡量方式 |
|------|--------|----------|
| 投票创建耗时 | ≤30 秒 | 从点击「发起投票」到发布成功的耗时 |
| 投票参与率 | ≥80% | 已投票人数 / 团队总人数 |
| 结果首屏加载 | ≤500ms | ECharts 渲染完毕的感知时间 |
| 实时推送延迟 | ≤2s | 他人投票后本端图表更新的时间差 |

### 1.3 范围定义

- **MVP 范围**：全团队可见投票，单选/多选，匿名/实名可选，倒计时/手动结束，实时结果
- **范围外（本期不做）**：定向投票人、改票、评论/讨论区、投票模板库、多轮次投票、导出 Excel、企业微信/钉钉适配

## 五、线框说明

> 以下为文字线框描述，供前端开发（流光）还原交互。

### 5.1 投票列表页 `/votes`

**布局**：单列信息流，顶部固定搜索/筛选栏，下方卡片列表。

**顶部栏**：
- 左侧：「团队投票」标题（字号 20px Bold）
- 右侧：「发起投票」主按钮（品牌色填充，圆角 8px）

**筛选 Tab**：
- 「进行中」(默认) | 「已结束」
- Tab 切换带下划线动效

**投票卡片**（每条）：
- 左侧状态指示点：绿色脉冲 = 进行中，灰色 = 已结束
- 卡片主体：
  - 标题（16px Medium，最多 2 行截断）
  - 标签行：`单选/多选` | `匿名/实名` | `剩余 05:30`
  - 进度条：「已投 12/24 人」+ 百分比进度条
- 点击整张卡片 → 进入 `/votes/:id`

**空状态**：
- 「暂无进行中的投票」插画 + 文案 + 「创建第一个投票」CTA 按钮

**加载中**：
- 骨架屏：3 张卡片骨架闪烁

### 5.2 创建投票页 `/votes/new`

**布局**：居中表单，最大宽度 560px，背景遮罩层（半屏弹窗或新页面，视飞书容器能力决定）。

**表单字段（自上而下）**：

1. **投票标题**
   - 单行输入框，placeholder「输入投票主题，如：Sprint 24 团建去哪儿？」
   - 字数限制 100 字，右下角实时计数 `12/100`
   - 必填，失焦校验非空

2. **选项列表**
   - 默认展示 2 个选项输入框
   - 每个选项：文本输入 + 右侧删除图标（选项数 >2 时出现）
   - 底部「+ 添加选项」文字按钮（选项数 <10 时可见）
   - 最少 2 个、最多 10 个选项，所有选项均需填写且不重复
   - 字数限制每选项 50 字

3. **投票设置区**（卡片背景，与上方表单分栏）
   - **投票类型**：单选 / 多选 — 分段控制器（Segmented Control）
     - 多选时出现辅助文案：「参与者可选择一个或多个选项，不设上限」
   - **投票模式**：匿名 / 实名 — 同分段控制器，默认「匿名」高亮
     - 选择「匿名」时下方出现蓝色提示条：「📌 匿名投票下，其他参与者看不到你的身份，但系统会记录你的投票以防重复提交」
   - **截止时间**：下拉或步进器选择分钟数
     - 预设选项：5分钟 / 15分钟 / 30分钟 / 1小时 / 自定义
     - 自定义弹数字键盘输入分钟数（1-10080，即最长 7 天）

4. **底部操作栏**（固定在表单底部）
   - 「取消」文字按钮（返回上一页 / 关闭弹窗）
   - 「发布投票」主按钮（品牌色，填充式）
     - 表单未填完时置灰不可点击
     - 点击后 loading 旋转 + 「发布中…」文案，成功跳转到 `/votes/:id`

**校验规则**：
- 标题非空（提交时校验，弹出 toast「请填写投票标题」）
- 每个选项非空且不重复（失焦校验，重复项标红 + 下方提示「选项不可重复」）
- 选项数 ≥2（<2 时不可发布）

### 5.3 投票详情页 `/votes/:id` — 进行中

**布局**：单列垂直流，移动端适配（最大宽度 640px 居中）。

**页面结构（自上而下）**：

1. **顶部信息栏**
   - 标题（20px Bold）
   - 元数据行：`单选 · 匿名 · 剩余 12:30` （标签形式，灰色背景圆角标签）
   - 进度：`已投 8/24 人` + 进度条（品牌色填充）

2. **选项列表**（未投票用户）
   - 选项卡片列表，每项包含：
     - 圆形单选按钮或多选方框（取决于投票类型）
     - 选项文本
     - 点击整行 → 选中/取消选中
   - 提交按钮：「提交投票」（全宽，品牌色）
     - 未选中任何选项时置灰不可点击
     - 多选模式下至少选 1 项
     - 点击后 loading → 成功变为「✓ 已投票」+ 选项锁定 UI

3. **已投票用户视图**
   - 选项恢复为只读展示，已选项目标色高亮 + ✓ 图标
   - 选项行不可点击
   - 底部提示：「投票已提交，不可更改」（灰色小字，带 info 图标）

4. **实时结果区**（所有用户可见，无论是否已投）
   - 标题：「实时结果」+ 实时更新时间戳 「刚刚更新」
   - ECharts 横向柱状图：
     - X 轴 = 票数，Y 轴 = 选项文本
     - 每根柱子：品牌色填充，末端显示票数数字
     - 柱右侧显示百分比（如 `37.5%`）
     - 自己投的选项柱颜色略深以示区分（仅本端可见）
   - 匿名模式下：无任何用户头像/姓名信息
   - 实名模式下：每个柱子右侧或 hover 可展开投票人列表（头像 + 姓名）
   - **乐观更新**：用户提交后本地立即 +1 渲染，不等待服务端确认；WS 推送其他用户投票时增量更新

5. **发起者专属操控区**（仅发起者可见）
   - 「结束投票」按钮（次危险样式，描边红色）
   - 点击弹出确认弹窗：「确定结束投票吗？结束后所有人不可再投票。」
     - 「取消」(灰色) / 「确认结束」(红色)
   - 确认后投票状态变更为「已结束」

**倒计时提醒**：
- 剩余 ≤ 60 秒：计时器变红 + 脉冲动画
- 剩余 ≤ 10 秒：计时器大号红色闪烁
- 剩余 0：页面自动切换为「已结束」状态

**WS 断线处理**：
- 断线时图表区出现顶部黄色横幅「网络连接中断，数据可能不是最新」
- 重连后横幅消失，全量拉取当前投票状态重新渲染

### 5.4 投票详情页 `/votes/:id` — 已结束

**布局**：与「进行中」页同结构，差异如下：

1. **顶部状态标签**：「已结束」灰色标签替代倒计时
2. **选项区域不显示投票交互**，仅展示最终结果柱状图
3. **柱状图区**：
   - 标题改为「最终结果」
   - 若为实名模式：展开显示各选项投票人明细（头像列表 + 姓名，不可折叠）
   - 总票数统计：「共 24 人投票，总票数 24」
4. **发起者视图**：无「结束投票」按钮，改为「重新发起」（次级按钮，灰度较低，本期可做可不做）
5. 页面底部文案：「投票已结束 · 由 [发起者姓名] 发起」

## 六、验收标准

> 验收标准采用三级标注：**正常路径** / **边界条件** / **异常路径**，供寻错编写测试用例。

### F-001 创建投票

| 编号 | 验收场景 | 前置条件 | 操作步骤 | 预期结果 |
|------|----------|----------|----------|----------|
| AC-001-1 | **正常**：创建单选实名投票 | 用户已登录，具有团队身份 | 填写标题「团建投票」→ 添加 3 个选项 → 选择「单选」「实名」→ 截止时间选「30分钟」→ 点击发布 | 投票发布成功，跳转至详情页，频道中出现投票卡片；选项数为 3，类型为单选，模式为实名，截止时间为当前时间+30分钟 |
| AC-001-2 | **正常**：创建多选匿名投票（默认匿名） | 用户已登录 | 打开创建页 → 默认显示「匿名」高亮 → 切换为「多选」→ 添加 5 个选项 → 不修改匿名设置 → 发布 | 发布成功，投票模式为匿名，多选 |
| AC-001-3 | **边界**：选项数=2（最小值） | 创建页打开 | 保留默认 2 个选项并填写内容 → 填写标题 → 发布 | 发布成功，选项数为 2 |
| AC-001-4 | **边界**：选项数=10（最大值） | 选项已添加至 10 个 | 点击「+ 添加选项」 | 「+ 添加选项」按钮不可见或置灰不可点击 |
| AC-001-5 | **边界**：标题长度=100 字（上限） | 创建页打开 | 输入 100 字标题 | 输入成功，不再接收更多字符，计数器显示 `100/100` |
| AC-001-6 | **边界**：截止时间为 1 分钟（最小值） | 创建页打开 | 自定义截止时间输入 1 | 输入成功，发布后倒计时从 01:00 开始 |
| AC-001-7 | **异常**：标题为空提交 | 创建页打开，标题未填 | 填写选项 → 点击「发布投票」 | 按钮不可点击（置灰），或点击后顶部 toast 提示「请填写投票标题」 |
| AC-001-8 | **异常**：选项有重复 | 创建页，选项 1 和选项 3 内容相同 | 失焦选项 3 或点击发布 | 重复的选项输入框标红，下方出现红色提示「选项不可重复」，不可发布 |

| 维度 | 要求 |
|------|------|
| 身份认证 | 所有 API 需验证飞书登录态（token/session），未登录拒绝请求 |

| 敏感数据 | 匿名投票下，API 返回给前端的投票人列表字段必须置空/null，不可泄露 user_id 映射 |
| **速率限制** ⭐新增 | 每人每分钟最多 3 次投票提交（不同投票独立计数，即同一用户对不同 vote_id 的提交分别计数），超过限制返回 HTTP 429 并附带 Retry-After 头；建议使用 Redis 滑动窗口计数器实现 |

## 九、API 接口概要 ⭐新增（R-01）

> 供栖梧做接口设计与前后端对齐，列出 MVP 阶段全部 REST 端点及 WS 事件。

### 9.1 REST 接口

| 接口名称 | 方法 | 路径 | 入参概要 | 出参概要 |
|----------|------|------|----------|----------|
| 创建投票 | POST | `/api/votes` | `{ title: string, options: string[], vote_type: 'single' \| 'multi', vote_mode: 'anonymous' \| 'public', deadline_minutes: number }` | `{ code: 0, data: { vote } }` — vote 对象含 id、title、status、deadline、options 等 |
| 投票列表 | GET | `/api/votes?status=active&page=1&size=20` | `status: 'active' \| 'closed'`, `page`, `size` | `{ code: 0, data: { items: vote[], total: number, page: number, size: number } }` |
| 投票详情 | GET | `/api/votes/:id` | —（路径参数 vote_id） | `{ code: 0, data: { vote, options[], tally (各选项当前票数), has_voted (当前用户是否已投) } }` |
| 提交投票 | POST | `/api/votes/:id/vote` | `{ option_ids: string[] }` — 至少 1 个选项 ID，单选时数组长度 =1 | `{ code: 0 }` 成功；`{ code: 409, message: '已投过票' }` 重复；`{ code: 403, message: '投票已结束' }` 已结束；`{ code: 429 }` 速率限制 |
| 结束投票 | POST | `/api/votes/:id/close` | —（仅发起者可调用） | `{ code: 0, data: { status: 'closed', closed_by: 'manual', closed_at } }` |

### 9.2 WebSocket 事件

| 事件名称 | 方向 | 触发时机 | Payload |
|----------|------|----------|---------|
| `vote:{id}:update` | 服务端 → 客户端 | 任一用户提交投票后广播 | `{ option_id: string, new_count: number, total_votes: number }` |
| `vote:{id}:closed` | 服务端 → 客户端 | 投票结束（手动或自动）后广播 | `{ closed_by: 'manual' \| 'auto', closed_at: timestamp }` |
| `vote:{id}:reminder` | 服务端 → 客户端 | 截止前 1 分钟 | `{ remaining_seconds: 60 }` |

### 9.3 通用约定

- **认证**：所有请求需携带飞书 SSO token（Header: `Authorization: Bearer <token>`）
- **响应格式**：统一 `{ code: number, message?: string, data?: any }`，code=0 表示成功
- **分页**：列表接口默认 `page=1, size=20`，最大 size=100
- **错误码**：`401` 未登录、`403` 无权限、`404` 资源不存在、`409` 业务冲突（如重复投票）、`429` 速率限制

## 十、约束与假设

### 技术约束 ⭐新增（R-02）

#### 自动结束定时器方案

投票倒计时到期后的自动结束逻辑，**必须持久化到外部存储，严禁使用纯内存 setTimeout**（进程崩溃会导致定时器丢失，投票永远不会自动结束）。推荐以下两种方案：

| 方案 | 实现方式 | 优点 | 缺点 |
|------|----------|------|------|
| **方案 A：Redis Keyspace Notification**（推荐） | 创建投票时向 Redis 写入带 TTL 的 key（`vote:{id}:deadline`，过期时间 = 截止时间）；Redis 配置 `notify-keyspace-events Ex`；服务订阅 `__keyevent@*__:expired` 通道，收到过期事件后触发结束逻辑 | 零依赖新增组件；Redis 本身已在架构中；TTL 由 Redis 自身维护，无需额外调度 | 需确保 Redis 开启 keyspace notification；过期事件有 ≤60s 延迟（Redis 惰性清理特性） |
| **方案 B：BullMQ 延迟任务** | 创建投票时向 BullMQ 提交延迟任务（`delay = deadline - now`）；到期后 worker 执行结束逻辑 | 精确到秒级触发；任务状态可观测 | 新增 BullMQ + Redis 依赖；需额外配置 worker 进程 |

**推荐方案 A**，理由：Redis 已在架构中使用（票数缓存），无需新增组件；TTL 过期机制天然适合「到时触发」语义。若实际测试发现 Keyspace Notification 延迟不可接受（>5s），则升级为方案 B。

**注意事项**：
- 服务重启时，需扫描 PG 中 `status='active' AND deadline < NOW()` 的投票，批量执行结束逻辑（兜底恢复）
- 同时保留「发起者手动结束」路径（优先级高于自动），手动结束时需主动删除 `vote:{id}:deadline` key 防止重复触发
- 自动结束后的 closed_by 字段设为 `'auto'`

### 约束
- 嵌入飞书环境，需使用飞书 SSO 登录态
- MVP 阶段仅支持全团队可见，不支持指定投票人
- 提交后不可改票（与老板决策一致）
- WS 推送依赖 Socket.IO 粘性会话（蓝绿部署场景需额外配置）

### 假设
- 团队规模 ≤500 人（MVP 验证范围）
- 用户网络环境为办公网络或 4G/5G（非极端弱网）
- 飞书客户端 WebView 内核版本支持 ECharts 5.x
- 团队信息可从飞书组织架构 API 获取

### 范围外（本期不做）
- 定向投票人（指定参与者范围）
- 改票功能
- 投票评论/讨论区
- 投票模板库
- 多轮次投票
- 导出 Excel/CSV
- 钉钉/企业微信适配
- 投票分享到外部
- 邮件/推送通知（超出 MVP 范围）

## 十一、风险评估 ⭐修订（R-05）

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 蓝绿部署导致 WS 粘性会话断裂 | 高 — 投票中断、数据不同步 | 中 | 提前与长夜确认负载均衡器配置；灰度阶段压测蓝绿切换场景 |
| 并发投票导致 Redis 计数竞争 | 中 — 票数不准 | 低 | Redis HINCRBY 原子操作天然防竞争；对账脚本定期校验 Redis 与 PG 一致性 |
| 匿名模式下 user_id 泄露 | 高 — 隐私合规风险 | 低 | API 层做字段级过滤；安全测试用例覆盖匿名模式 API 返回值 |
| 倒计时依赖客户端渲染，时钟不准 | 中 — 抖动或提前结束 | 中 | 以服务端时间戳为准，WS 推送归零事件；客户端仅供展示 |
| 飞书 WebView 兼容性未知 | 中 — ECharts 渲染异常 | 低 | 提前在飞书桌面端+移动端内部测试版做兼容性验证 |
| **服务进程崩溃导致内存定时器丢失** ⭐新增 | 高 — 投票永远不会自动结束，状态永久卡在「进行中」 | 低 | 定时器持久化到 Redis/PG（自动结束方案见第十章技术约束）；服务重启时兜底扫描 PG 中到期未结束的投票 |
| **团队人员变更导致 total_voters 不一致** ⭐新增 | 中 — 参与率统计（已投/总人数）失真，可能超过 100% 或低于实际 | 中 | `total_voters` 标注为「创建投票时刻的快照值」，前端展示时附带提示「创建时团队人数」；参与率仅作参考不强制校验 |
| **Redis 不可用** ⭐新增 | 高 — 票数缓存丢失、速率限制失效、定时器无法触发 | 低 | 降级为直接读写 PG（见 7.2 安全·Redis 降级）；配置 Redis 连接健康检查 + 飞书告警；降级期间功能可用但性能下降 |
| **ECharts 在低端移动设备渲染性能** ⭐新增 | 中 — 柱状图卡顿、动画掉帧、交互延迟 | 中 | 提前在飞书移动端低端机型（如 iPhone SE 1st gen / 入门安卓机）做基准测试；必要时关闭动画或降级为纯 CSS 柱状图（无 Canvas） |
