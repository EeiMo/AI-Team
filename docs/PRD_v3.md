## 二、模块 M1：创建人删除投票

### 2.1 现状痛点

| 痛点 | 详情 | 影响 |
|------|------|------|
| 创建者无法删除 | 投票一经创建无法撤销，即使创建者发现拼写错误、选项不合理 | 创建者只能等待投票结束或被用户"忽略"，体验割裂 |
| 无数据软删除能力 | 数据库 votes 表无 `del_flag` 字段，所有删除即为物理删除 | 不符合审计要求，无法追溯已删除的投票记录 |
| Redis 残留 | 即使未来支持物理删除，仍需级联清理 Redis tally、deadline key、WS room | 数据残留可导致脏数据和状态不一致 |

### 2.2 解决方案总览

采用**软删除**方案：
- 数据库 `votes` 表新增 `del_flag CHAR(1) DEFAULT 'N'` 字段（`'N'`=未删，`'Y'`=已删）
- 删除后数据保留不丢失（支持恢复/审计查询）
- 已删除投票在所有前端列表中不可见（实时同步移除）
- 已删除投票的投票详情页显示「该投票已被创建者删除」替代页面
- 删除操作写入审计日志（记录操作人、操作时间、被删投票 ID）

### 2.3 删除允许状态

| 投票状态 | 是否可删除 | 说明 |
|----------|-----------|------|
| `active`（进行中） | ✅ 可删除 | 删除后投票即时终止，已投数据保留，Redis 清理 |
| `closed`（已结束） | ✅ 可删除 | 删除后结果页替换为「已删除」状态提示 |
| `deleted`（已删除） | ❌ 不可重复删除 | 幂等性要求：第二次调用返回 404 或明确提示 |

### 2.4 删除后各模块行为

| 模块 | 行为 |
|------|------|
| **PostgreSQL** | `votes.del_flag` 更新为 `'Y'`（软删除） |
| **Redis tally** | 删除 `vote:{id}:tally` hash，删除 `vote:{id}:deadline` key |
| **WS 推送** | 广播 `vote:{id}:deleted` 事件到该投票的 WS room |
| **前端** | 收到 deleted 事件后：①列表页移除该投票卡片（带过渡动效）；②详情页若当前在此页面则显示「已删除」占位界面 |
| **审计日志** | 插入审计记录表：`action='delete_vote'`，记录 `user_id`/`vote_id`/`timestamp`/`ip`/`user_agent` |
| **其他用户** | 已投此投票的其他用户在投票历史和「我的投票」中仍可见投票旧数据，但详情页显示「已删除」 |

### 2.5 删除流程（时序）

```
[创建者]
    │
    │ 点击列表页/详情页「删除」按钮
    ▼
[前端确认弹窗] → 文案：「确定删除投票「投票标题」吗？删除后数据仍保留在系统内，但所有用户将无法再查看和投票。此操作不可撤销。」
    │
    │ 确认
    ▼
[DELETE /api/votes/:id] (携带 JWT token)
    │
    ▼
[后端鉴权] → 校验：
  1. JWT 解析 → 获取当前 user_id + team_id
  2. votes 表查询 → 校验 vote.creator_id == current_user_id
  3. 校验 vote.team_id == current_user_team_id (双重鉴权)
  4. 校验 vote.del_flag != 'Y' (幂等)
    │
    ▼
[执行软删除]
  1. UPDATE votes SET del_flag='Y', deleted_at=NOW(), deleted_by=current_user_id WHERE id=:vote_id AND del_flag='N'
  2. DELETE vote:{id}:tally (Redis)
  3. DELETE vote:{id}:deadline (Redis)
  4. WS emit('vote:{id}:deleted', { vote_id, deleted_by, deleted_at })
  5. INSERT INTO audit_logs (action, entity_type, entity_id, user_id, team_id, ip, user_agent, created_at)
    │
    ▼
[返回前端] → 200 { code: 0, message: '投票已删除' }
    │
    ▼
[前端处理]
  1. 当前页为列表页：卡片以淡出动画移除
  2. 当前页为详情页：页面内容替换为「该投票已被创建者删除」界面
  3. 其他在线用户：收到 WS → 列表页移除卡片/详情页替换内容
```

## 三、模块 M2：前端美工优化

### 3.1 背景与原则

**背景**：v1.2 上线上线后，产品视觉效果有提升空间。老板要求提升「高级感」，但「美化」作为主观目标，必须在本 PRD 中卡死实际改动范围，杜绝无尽的 scope 蔓延。

**核心原则**：
1. **不引入第三方 UI 库** — 仅使用现有 CSS/React 技术栈进行优化
2. **按页面粒度、组件粒度锁定改动** — 每项改动有明确的验收标准
3. **不加新功能** — 不新增页面、不新增交互流程、不新增按钮/区块
4. **移动端优先** — 优化以移动端+飞书内嵌 WebView 为主，桌面端保持对齐

### 3.2 优化页面/组件粒度清单

| 编号 | 优化对象 | 优化内容 | 优先级 |
|------|---------|----------|--------|
| U-01 | **全局色彩系统** | 统一品牌色、辅助色、状态色（成功/警告/危险/禁用）CSS 变量体系；确保所有页面使用同一套色板 | P1 |
| U-02 | **全局动效基调** | 定义全局过渡动画标准（页面切换/卡片出现/按钮状态切换），统一 transition 时长和缓动函数 | P1 |
| U-03 | **实时计票结果展示（重点）** | 投票详情页「实时结果」区块重新设计：卡片布局提升品质感、票数变化动效（数字跳动过渡）、柱状图动画平滑、进度条从单调变为渐变/圆角 | **P1 重点** |
| U-04 | **投票列表页卡片** | 卡片阴影/圆角统一、状态指示点脉冲动效、标签样式统一、进度条样式升级 | P1 |
| U-05 | **创建投票页表单** | 输入框聚焦态样式增强、标签/控件区域留白优化、断行/校验错误提示样式统一 | P1 |
| U-06 | **空状态/错误状态** | 优化空白/错误/加载中的视觉占位：统一的插画风格、文字居中布局、CTA 按钮样式统一 | P1 |
| U-07 | **投票详情页布局微调** | 顶部信息栏卡片化、元数据标签样式升级、「已投票」状态区增强；匿名隐私提示条样式调整 | P1 |
| U-08 | **移动端专项适配** | 确认 320px~414px 范围内各组件样式不溢出/不错位；点击区域大小符合触控规范（≥44px） | P1 |
| U-09 | **删除操作交互（M1 联动）** | 「删除」按钮样式（危险操作风格）、确认弹窗样式美化、删除后列表淡化移除动效 | P1 |

### 3.3 重点优化：实时计票展示 UI（U-03 详细设计）

这是老板指定的重点优化区域，下面给出详细设计方向。

#### 当前问题
- 柱状图直接嵌入页面，缺少容器和留白，视觉上较"简陋"
- 票数数字随着 WS 更新直接跳变，无过渡感
- 进度条单一颜色，识别度不高
- 整体布局与页面其他元素边界模糊

#### 优化方向

**3.3.1 卡片式结果容器**
- 将实时结果区包裹在独立卡片中：圆角 12px、浅色背景（var(--bg-card)）、轻微阴影
- 卡片顶部：标题「实时结果」左对齐 + 更新时间戳「刚刚更新」右对齐
- 卡片内部：上下左右 16px 内边距

**3.3.2 票数变化动效（重点中的重点）**
- 每个选项的票数数字在 WS 推送更新时，从旧值平滑过渡到新值（数字滚动/闪烁效果，类似 macOS 计时器数字过渡）
- 选项卡片在新票数到达时，有微弱的「脉冲」高亮（250ms 淡入 + 250ms 淡出）
- 柱状图柱子宽度变化带 CSS transition（宽度变化 300ms ease-out）

**3.3.3 进度条升级**
- 从纯色扁条改为渐变填充 + 圆角两端 + 轻微内阴影
- 进度条上方或内部显示百分比文字
- 已有投票当前选项的柱子显示不同的色调（本端染色，体现「我已投这个」）

**3.3.4 全局动效基调**
- 所有状态切换（选中/未选中、加载/完成、投票前/投票后）统一过渡 200-300ms ease
- 不追求复杂的 keyframe 动画或粒子效果，以简洁平滑为主

### 3.4 禁止项（美化范围护栏）

以下不在本期美化范围内，如发现 PR 包含以下内容，直接打回：

| 禁止项 | 说明 |
|--------|------|
| ❌ 引入第三方 UI 组件库 | 如 Ant Design、Material UI、Tailwind CSS、Chakra UI 等 |
| ❌ 升级/更换图表库 | ECharts 保持不变 |
| ❌ 新增页面或路由 | 仅优化现有 4 个页面：列表页/创建页/详情页(active)/详情页(closed) |
| ❌ 新增交互或功能 | 如投票动效粒子、用户头像动效、投票主题切换等 |
| ❌ 新增图标库 | 保持现有图标方案，不加 FontAwesome、Tabler Icons 等 |
| ❌ 字体更换 | 使用现有字体，不引入 Google Fonts/自定义字体 |
| ❌ 动画过度复杂化 | 不允许加入长时间（>500ms）或循环动画，避免性能下降和辣眼睛 |

## 六、交互原型 / 页面流

### 6.1 新增交互：删除流程

```
[投票列表页 /votes]
    │
    ├─ 每个投票卡片 → 新增「删除」入口
    │     (仅创建者可见，置于卡片右下角/底部操作区)
    │     (图标：🗑️ 垃圾桶，带危险色)
    │
    ├─ [点击删除]
    │     │
    │     ├─ 弹出确认弹窗
    │     │    ┌────────────────────────────┐
    │     │    │  ⚠️ 确认删除投票             │
    │     │    │                              │
    │     │    │  投票标题：「Sprint 24 团建  │
    │     │    │  去哪儿？」                    │
    │     │    │                              │
    │     │    │  删除后数据仍保留在系统内，     │
    │     │    │  所有用户将无法再查看和投票。   │
    │     │    │  此操作不可撤销。              │
    │     │    │                              │
    │     │    │    [取消]  [确认删除]         │
    │     │    └────────────────────────────┘
    │     │
    │     └─ 确认 → 调用 DELETE API
    │           │
    │           ├─ 成功 → 卡片淡出动效移除
    │           │          toast「投票已删除」
    │           │
    │           └─ 失败 → toast 错误信息

[投票详情页 /votes/:id (仅创建者)]
    │
    ├─ 发起者专属操控区 → 新增「删除」按钮
    │   (位于「结束投票」按钮旁或下方，危险样式)
    │
    └─ [点击删除] → 同上确认弹窗流程
         │
         └─ 成功 → 页面内容替换为「已删除」占位页
                     (包含图标/文案/「返回列表」按钮)

[其他在线用户收到 WS]
    │
    ├─ 列表页 → 卡片淡出动效移除
    └─ 详情页(正在查看此投票) → 页面替换为「已删除」占位页
```

### 6.2 新增页面/状态

| 状态 | 路由/展示方式 | 说明 |
|------|-------------|------|
| 已删除占位页 | `/votes/:id`（同路由，状态驱动） | 投票已被创建者删除时的替换页面，包含 🗑️ 图标、文案「该投票已被创建者删除」、按钮「返回列表」 |

### F-301 删除投票

| 编号 | 验收场景 | 前置条件 | 操作步骤 | 预期结果 |
|------|----------|----------|----------|----------|
| AC-301-1 | **正常**：创建者删除进行中的投票 | 用户 A 是投票 1 的创建者；投票 1 状态为 `active` | 用户 A 在列表页点击投票 1 卡片上的删除 → 确认弹窗 → 点击「确认删除」 | ①返回 200，`del_flag` 更新为 Y；②Redis `vote:{id}:tally` 被删除；③Redis `vote:{id}:deadline` 被删除；④WS 广播 `vote:{id}:deleted`；⑤审计日志插入一条 delete_vote 记录；⑥前端 toast「投票已删除」 |

| AC-301-10 | **异常**：Redis 不可用时删除 | Redis 服务停止 | 用户 A 执行删除 | ①DB 软删除成功（del_flag=Y）；②Redis 清理失败但**不影响主流程**，记录错误日志；③返回 200，前端正常更新（Redis 为辅助缓存，不可用时不可阻塞主流程） |
| AC-301-11 | **异常**：投票已被其他人投票中，创建者删除 | 投票 active，有 5 人已投票 | 创建者删除 | 删除成功，已投数据保留（user_votes 表不变），删除后其他用户的详情页变为「已删除」状态 |

## 八、非功能需求

### 8.1 性能

| 指标 | 要求 | 说明 |
|------|------|------|
| 删除 API 响应（P99） | ≤500ms | 含 DB UPDATE、Redis DEL(×2)、WS emit、审计日志插入 |
| 美化后页面 FCP | ≤1.5s（不变） | 新增 CSS 动画不显著提升渲染时间 |
| 动效帧率 | ≥30fps（低端移动端）、≥60fps（桌面端） | 动效不得导致明显卡顿 |

### 8.2 安全

| 维度 | 要求 |
|------|------|
| 删除鉴权 | 必须 `creator_id` + `team_id` 双重校验（同团队内也禁止非创建者删除） |
| 幂等性 | 已删除投票的重复删除请求不产生副作用 |
| 审计可追溯 | 所有删除操作必须写审计日志，不可跳过 |
| 防 CSRF | DELETE API 需要有效 JWT token
| 删除前后端校验 | 前端仅显示删除入口给创建者，但后端仍须独立鉴权（前端约束不可替代后端安全校验） |

### 8.3 兼容性

| 维度 | 要求 |
|------|------|
| 浏览器 | Chrome 90+ / Edge 90+ / Firefox 90+ / Safari 15+（不变） |
| 飞书客户端 | 飞书桌面端 ≥6.0 / 飞书移动端 ≥6.0（不变） |
| 屏幕尺寸 | 移动端 320px 起，桌面端 640px 居中（已覆盖 M2 优化需求） |

### 8.4 可用性

| 维度 | 要求 |
|------|------|
| 数据持久性 | 软删除保留所有历史数据，无物理删除 |
| 故障恢复 | Redis 不可用时，删除操作仍可完成 DB 更新（以 DB 成功为准，Redis 清理失败记录日志）；服务重启后已删除投票仍保持已删除状态 |

### 8.5 可观测性

| 维度 | 要求 |
|------|------|
| 审计日志 | 删除操作必须写入独立审计日志表/日志流 |
| 错误日志 | Redis 清理失败、WS 推送失败均记录日志 |
| 指标 | 删除操作的 QPS、成功率 P99、审计日志写入延迟 |

## 十、API 变更

### 10.1 新增 REST 接口

| 接口名称 | 方法 | 路径 | 入参概要 | 出参概要 |
|----------|------|------|----------|----------|
| 删除投票 | DELETE | `/api/votes/:id` | —（路径参数 vote_id；需 JWT header） | `{ code: 0, message: '投票已删除' }` 成功；`403` 无权限；`404` 不存在或已删除 |

### 10.2 现有接口变更

| 接口名称 | 变更 |
|----------|------|
| GET `/api/votes?status=...` | 默认过滤 `del_flag='N'`（不返回已删除投票）；如需查询全部（审计用途）需额外参数 `?include_deleted=true` |
| GET `/api/votes/:id` | 已删除投票返回 `code` 非 0，`data` 中携带 `deleted: true`、`deleted_at`、`deleted_by` 信息，前端据此渲染「已删除」占位页 |
| POST `/api/votes/:id/close` | 对已删除投票调用返回 404 |
| POST `/api/votes/:id/vote` | 对已删除投票调用返回 403「投票已删除」 |

### 10.3 新增 WebSocket 事件

| 事件名称 | 方向 | 触发时机 | Payload |
|----------|------|----------|---------|
| `vote:{id}:deleted` | 服务端 → 客户端 | 创建者执行删除后广播 | `{ vote_id: string, deleted_by: string, deleted_at: timestamp }` |

### 10.4 现有接口返回值调整的影响

- 前端投票列表 API 调用需适配：若返回值已不过滤 del_flag='N'，需后端默认过滤
- 前端投票详情页需处理 `deleted: true` 的分支，渲染「已删除」占位页
- 所有 GET 列表接口分页计数应考虑已删除过滤

## 十一、约束与假设

### 技术约束

1. **DB 变更**：所有数据库变更必须通过 migration 脚本执行，严禁手动修改线上数据库
2. **Redis 降级不影响主流程**：Redis 不可用时，删除操作仍应以 DB 成功为准；Redis 清理失败记录错误日志
3. **WS 事件现有模式**：`vote:{id}:deleted` 事件命名遵循现有 `vote:{id}:*` 命名约定（参考 v1.1 的 update/closed/reminder 事件）
4. **CI/CD 人工验收**：本轮不修复 CI/CD 流水线，所有变更通过人工验收确认

### 假设

- 投票创建者 `creator_id` 与飞书用户 ID 一致（飞书 SSO 已提供真实身份）
- 所有已存在的投票记录（v1.1+v1.2 创建的）默认 `del_flag = 'N'`（数据兼容）
- 前端现有 CSS 改造仅基于现有技术栈（CSS Modules 或 styled-components，无框架切换）
- 用户 `user_agent` 和 `ip` 在请求头中可获取

### 范围外（本期不做）

- CI/CD 修复/跑通
- 数据库架构重构
- 引入第三方 UI 组件库
- 新增页面或路由
- 改票功能、多轮次投票
- 投票模板库、导出 Excel/CSV
- 钉钉/企业微信适配
- 飞书组织架构同步
- 超过 500ms 的复杂动画或粒子效果

## 十二、风险评估

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 删除操作 Redis 清理遗漏（tally/deadline 清理不完整） | 高 — 数据残留、WS room 残留 | 中 | AC-301-1 明确验证 Redis 清理；寻错测试用例全覆盖 Redis key 清理验证 |
| 删除后 WS 推送失败导致部分用户不及时感知 | 中 — 用户看到已删除投票仍在列表中 | 低 | WS 推送失败不影响主流程；用户刷新页面后列表不再显示（API 层已过滤） |
| 美化范围 scope 蔓延 | 中 — 工期超支、未预期变更多 | 中 | PRD 3.4 节明确定义「禁止项」清单（6 条），作为评审门禁直接打回依据 |
| 移动端动效/动画性能问题 | 中 — 低端设备卡顿 | 中 | AC-306-5/6 规定动效时长 ≤500ms 且无循环动画（状态指示点除外）；8.1 节规定动效帧率 ≥30fps；前端开发需在低端设备上实测 |
| 审计日志增长过快 | 低 — 小型项目审计量有限 | 低 | 当前阶段不设自动清理策略，仅记录删除操作（非每次投票提交），增长可控 |
| 删除操作被恶意批量调用 | 中 — 批量投毒 | 低 | 删除 API 已在鉴权层通过 JWT 约束，仅创建者可调用；如需要，可增加速率限制 |

### B. 术语表

|------|------|
| 软删除 | 标记 `del_flag='Y'` 而非物理删除数据行，记录 retained 状态 |
