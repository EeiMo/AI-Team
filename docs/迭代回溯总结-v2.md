## 三、Top 3 问题与根因分析

### 🥇 问题一：CI/CD 流水线从搭建到完成始终未跑通（阶段四/五卡死）

**根因分析**：
- **Runner 注册依赖人工操作**：`./config.sh` 需要 GitHub registration token，长夜无法自动完成注册
- **多个配置问题连环打击**：ESLint 未安装 → config.ts ESModule 冲突 → Test 缺少 npm ci → Jest Babel 解析 `as const` 失败 → 每次修复需要等一次完整 workflow run
- **无本地 CI 模拟机制**：每次修改后无法本地模拟 runner 环境快速验证，只能 push → 等 GitHub Actions 跑（5-10 分钟一轮）
- **snap Docker 兼容性问题**：OpenClaw sandbox 无 Docker Engine，部署脚本无法在 sandbox 中验证

**相关各方共识**：长夜、凌霜、流光、云起均提到同样的问题链

### 🥈 问题二：「投票创建者可删除投票」功能未经评审即实现并推送（阶段六后）

**根因分析**：
- **边界失守**：EeiMoo 在老板口头提及需求后，未汇报确认 scope 就直接修改并 push 到 main
- **无范围扫描门禁**：缺乏自动化的「是否在本期 PRD 范围内」的检查机制
- **后果**：功能在 30 分钟内被 revert，产生无效功（5 个文件改动 + 1 revert commit）

### 🥉 问题三：生产部署 SSH 认证失败 + 环境变量遗漏，依赖老板多次人工介入

**根因分析**：
- **生产环境准备工作不足**：SSH 公钥未预先注册到生产服务器
- **Docker Compose 环境变量传递不完整**：`JWT_SECRET`、`FEISHU_APP_ID` 等关键变量未通过 environment 段传递
- **部署方案缺少「老板协作清单」**：没有主动标注哪些步骤需要人工介入

## 四、改进措施与责任人

| 改进项 | 类型 | 责任人 | 目标 | 优先级 |
|--------|------|--------|------|--------|
| **CI 预检流程** — CI 配置变更后，先本地模拟 runner 环境验证再合入 | 流程规则 | 长夜 + 凌霜 | 减少「push→等 5 分钟→失败」的无效循环 | P0 |

| **ORM 类型兼容检查清单** — 数据库类型兼容性预先查验 | 知识缺口 | 凌霜 | 避免 UUID[] 类问题重犯 | P1 |
| **生产环境准备 Checklist** — SSH 公钥 / 环境变量 / 网络连通性前置检测 | 评审盲区 | 长夜 + 栖梧 | 生产部署不卡住 | P0 |
| **WS 事件模型文档化** — 明确 WS 事件触发时机/payload/处理策略 | 模板缺失 | 栖梧 + 流光 | 杜绝乐观更新数据不一致 | P0 |

## 六、自进化线索（阶段九输入）

本次回溯识别的主要进化触发源：
