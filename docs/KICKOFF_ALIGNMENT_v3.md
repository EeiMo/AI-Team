## 一、统一项目愿景

**一句话**：在稳定运行的投票系统上，以标准化迭代流程完成两个增量交付——允许创建人删除自己的投票，以及打磨前端 UI 品质。

**核心共识**：
- v3 是增量迭代，不涉及架构重构或技术栈更换
- 现有 Docker Compose + PG 15 + Redis 7 + Nginx + Express + React 基础设施保持不动
- 所有需求变更走正式评审流程，杜绝"未经评审直接合入"（v2 回溯教训）
- CI/CD 门禁缺失问题已识别，各角色一致认为应优先修复，但最终决策权在老板

## 三、风险预判（汇总·按严重度排序）

### 🔴 P0：删除操作的级联完整性

跨越 PG / Redis / WS / 前端的完整清理链条，任何遗漏都会导致数据残留。

**涉及**：凌霜（后端实现）、寻错（测试覆盖）、知微（安全校验）、长夜（可回滚迁移）

### 🔴 P0：删除权限校验

只有创建者可删；需要 `creator_id == current_user` 双重校验 + 仅允许指定 status 的投票删除。

**涉及**：凌霜、知微

### 🟡 P1：前端美化 scope 失控

"美化"是主观目标，若不卡死页面粒度和验收标准，容易变成无底洞。

**涉及**：云起（PRD 中卡死范围）、流光（执行）、老板（决策）

### 🟡 P1：v2 P0 漏洞回归风险

本次涉及后端代码修改（新增 DELETE 路由），可能改动 `middleware/auth.ts` 等关键文件，需确认 v2 P0 漏洞修复未被回退。

**涉及**：凌霜、知微

### 🟡 P1：删除后审计数据丢失

硬删除会让投票记录、用户投票记录彻底消失。软删除方案可保留审计数据。

**涉及**：知微、寻错

### 🟡 P1：CI/CD 依然不通

代码变更无任何自动化验证，直接 push main。如本轮无法修复，人工验收须更严格。

**涉及**：长夜（最终执行）、老板（决策）

### 🟢 P2：Redis 清理遗漏

tally hash、deadline key、幂等缓存的清理容易被遗忘。

**涉及**：凌霜、寻错（测试覆盖）

## 四、各角色定位与边界

### 📋 云起（PM）
- **该做**：转译模糊需求为结构化 PRD、画出交互流程与页面原型、确定验收标准
- **不该做**：写代码、替栖梧做技术选型、替长夜做运维方案、替寻错写测试用例、未经评审口头承诺功能

### 🏗️ 栖梧（架构）
- **该做**：评审 PRD（技术可行性初判）、输出 v3 架构增量方案（删除 API/状态机/数据流/安全设计/Redis 清理/WS 事件）、按回溯改进项穷举环境变量变更清单
- **需要输入**：①硬删/软删决策 ②前端美化范围内的架构约束需求 ③PRD 到位时间

### 💻 凌霜（后端）
- **该做**：DELETE API 接口设计与实现、数据库迁移脚本（含回滚）、接口文档、单元/集成测试+自测报告
- **不该做**：前端 UI 改动、DevOps/CD 配置、越过架构评审直接写代码

### 🎨 流光（前端）
- **该做**：删除投票的前端 UI/交互/API 对接、WS 监听新事件联动、前端美化（样式/动效/色彩一致性）
- **不该做**：后端删除接口设计、DB 策略决策、CI/CD 调整
- **需对齐**：云起确认美化范围 + 交互细节（列表页是否提供删除入口）

### 🔍 寻错（测试）
- **该做**：测试策略与用例（全功能/边界/异常）、删除 API 自动化全覆盖（含越权）、级联清理验证（PG+Redis）、前端交互测试、安全审查测试
- **不该做**：决定硬删/软删、制定美化视觉标准、实现代码

### 🚀 长夜（运维）
- **该做**：CI/CD 流水线维护、环境管理（dev/test/staging/prod）、部署方案评审、生产发布+回滚执行、运维回溯报告
- **受本迭代影响**：极小——新增 DELETE API 不涉及架构变更，流程无新增依赖。只需确认 DB migration 有回滚脚本

### 🛡️ 知微（数据安全）
- **该做**：安全初检（API 权限模型校验、代码安全审计、依赖漏洞扫描、敏感数据脱敏复查）、安全渗透测试（涵盖删除功能）、线上数据观察
- **底线要求**：①删除 API 必须 `creator_id + team_id` 双重校验 + 仅允许 closed 状态可删 + 硬删则清理 Redis + 备软删方案 ②v2 P0 漏洞修复不可被回退 ③删除操作必须有完整审计日志
