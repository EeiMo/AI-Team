# 后端开发自测通用清单

> 每次提交 PR 前必须完成本清单全部检查项。本清单是防御生产逃逸 bug 的最后一道防线，基于 3 个真实 P0 事故归纳。

---

## 1. 清空数据后的完整端到端路径

- [ ] **清空本地数据后完整端到端路径走通**

  **踩过的坑（P0-001）**：本地开发数据库有之前手动塞入的测试数据，某些查询走了已有数据的路径，看起来一切正常。生产环境空数据库部署后，首次真实请求触发了一个依赖"至少有一条记录"的 SQL 逻辑分支，直接报错。

  **正确做法**：
  ```bash
  # 1. 清空本地数据
  npx knex migrate:rollback --all
  npx knex migrate:latest

  # 2. 从头走完核心业务流程（不依赖任何预置数据）
  # - 注册 → 登录 → 创建投票 → 投票 → 查看结果
  ```

---

## 2. ≥3 种入参组合覆盖

- [ ] **≥3 种入参组合（正常/边界/异常）**

  **踩过的坑（P0-002）**：API 开发完成后只测了"快乐路径"（正常参数，正常流程），从未测试"用户传空字符串"、"选项 ID 不存在"、"重复投票"等情况。生产环境中某 API 在正常流量下没问题，但被安全扫描器用异常参数扫到时直接 500。

  **测试矩阵**：
  ```
  | 场景     | 示例入参                              | 预期结果        |
  |----------|--------------------------------------|----------------|
  | 正常     | 合法参数，正确流程                    | 200 / 正常返回  |
  | 边界     | 空值、超长字符串、极限数值            | 400 / 合理拒绝  |
  | 异常     | 不存在的 ID、过期 token、SQL 注入尝试 | 4xx / 不崩溃    |
  ```

---

## 3. 真实 PG 环境（非 mock/SQLite）

- [ ] **真实 PG 环境（非 mock/SQLite 内存库）**

  **踩过的坑（P0-003）**：本地开发用 SQLite 内存库跑测试（快），CI 也是 SQLite。但生产是 PostgreSQL。SQLite 对 `ALTER TABLE` 约束、`jsonb` 操作、`RETURNING` 子句的行为与 PG 不同。本地 100% 通过的测试，部署到 PG 后 3 个 API 全部报错。

  **正确做法**：
  - 单元测试可以用 mock（测试逻辑）
  - 集成测试 / 端到端自测**必须**连真实 PG
  - CI/CD 环境必须使用与生产一致的 PG 主版本（不能 PG 16 本地 → PG 14 生产）

---

## 4. 自测结果留存

- [ ] **自测结果截图/日志留存**

  **踩过的坑**：口头说"测过了"，没有留下任何证据。生产出问题后追溯，无法确认当时到底测了哪些场景。复盘会上变成了"我记得测了" vs "你那不算真正测过"的消耗战。

  **正确做法**：
  ```
  # 自测完成后生成一份自测报告
  echo "=== 自测报告 ===" > self-test-$(date +%Y%m%d-%H%M).log
  echo "PR: #123" >> self-test-$(date +%Y%m%d-%H%M).log
  echo "  [PASS] 清空数据 + 完整端到端 ✓" >> ...
  echo "  [PASS] 正常入参 ✓" >> ...
  echo "  [PASS] 边界入参 ✓" >> ...
  echo "  [PASS] 异常入参 ✓" >> ...
  echo "  [PASS] 真实 PG 环境 ✓" >> ...
  # 附截图或关键日志片段
  ```
